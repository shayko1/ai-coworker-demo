09:15

A bug lands. Copy this prompt.

The agent runs inside the repo: it reads code and runs tests. The ticket: SiteMetadataService throwing error 500 in prod. The weak move:

here is the stack trace: <paste>. fix it

That buys a confident guess. Instead:

getSiteMetadata returns 500 for ~5% of sites.
The handler is SiteMetadataService.getMetadata.
Expected: metadata response. Actual: NullPointerException.
Reproduce: msid 3f2a… fails every time. New sites work.

Before you change anything: give me your two best
suspects, and how we prove each one.

10:30

You fix it in four moves

Its first suspect was right: old sites are missing a field. The fix: fill it in, and record where each value came from.

01

Research

"Read the site-metadata flow. Don't change anything yet."

02

Plan

It proposes steps and risks. You approve the scope.

03

Implement

One step, small diff. Too many files? Split it.

04

Verify

"Run the tests. Show me the output." Green is the finish line.

Nothing moves without your OK.

12:40

You open the PR. Nine minutes later, a review.

It came from the team's PR-review agent, built because writing code got fast and waiting for review did not. One of its comments is about your diff.

You
Open the PR.
Agent
Picks it up. README change: light pass. Business logic: deep pass.
Agent
Reviews against the team's rules. Comments inline, on your lines.
You
Push a fix. It re-reviews just the new commits.
Team
Final review and the merge. The agent is the first pass, never the last.

Every team runs its own review agent; AI-infra now offers a central one. Check what your team uses.

12:49

The comment waiting on your diff

AIpr-review-bot commented on site_metadata.proto
You added site_source with the value TEMPLATE. But site_type, right above it, already has TEMPLATE. The same situation can now be written two ways. What prevents a contradictory combination, and which field wins if they disagree?

Two questions, not a verdict. And the first time this pattern showed up, the bot approved it.

lunch · where the rule came from

How did it know? It didn't. Not the first time.

# from our team's review skill. Real rule, names simplified.

### New field / enum: semantic-overlap check (ALWAYS-ON)

Trigger: a new field lands next to a similar one
(site_source next to site_type).

1. Read the existing field's FULL value list from master.
2. Build the overlap table yourself, value by value.
3. Do NOT just accept the author's "why not reuse X". Check it.

Canonical miss: PR #18874.

This document is a skill: team knowledge written down, read by the agent before every review.

13:20 · you improve the skill

A skill is a file. You can edit it.

Nobody retrains a model. Someone writes the lesson down.

01

Notice the repeat

You corrected the agent twice about the same thing.

02

Write it into the skill

Plain words, one example. Open a PR like any change.

03

Every teammate gets it

From the next run, every teammate benefits.

Your first contribution here can be a rule, not code.

14:30 · your turn to review

Today you're the reviewer

A teammate asks you to review their PR. Big diff, code you barely know.

Walk me through this PR: what changed, and why.
Which parts are risky? What do the tests not cover?
The review bot already commented. What did it miss?
Questions only. I make the call on each one.
  • It explains and flags. You judge.
  • The bot ran the checklist. You bring context and judgment.
  • You post the comments. Your name is on the review.

16:40

Then it gets one wrong

Your team lead asks how many sites the bug hit. The agent answers in seconds: a precise number, full confidence. It never ran a query. This is a hallucination: a confident guess dressed as a fact.

safe-to-lean-on

  • Tests, repeated code, small changes everywhere
  • Explaining code you don't know yet
  • First drafts of anything

double-check

  • Money, permissions, user data
  • Numbers it quotes: "show me the query"
  • Anything you can't run yourself

17:45

Support pings: "sent twice?"

A site owner says their campaign email went out twice. One command starts it: /investigation, our team's skill. The central system is prod-e, where every team plugs in its own.

01

Hand it the ticket

The link, the site id, one line: "sent twice?"

02

It triages first

Infra? User setup? Another team? Ours? Most tickets are not ours.

03

It digs

Logs, data, metrics, recent changes.

04

It reports as it goes

Every query and finding lands in a draft doc.

This one comes back: ours. The evidence is on the next slide.
